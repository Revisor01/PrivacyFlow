  <action>
    Das Problem: `updateAccountSites` (Zeile 136) ruft `updateWidgetCredentials` auf BEVOR `applyAccountCredentials` fertig ist. Das fuehrt zu einem vorzeitigen `reloadAllTimelines()` mit inkonsistenten Daten.

    Ausserdem: In `applyAccountCredentials` wird `updateWidgetCredentials` (Zeile 302) VOR `NotificationCenter.post(.accountDidChange)` (Zeile 305) aufgerufen. Das ist korrekt — aber `updateAccountSites` ist der eigentliche Problemverursacher.

    **Fix 1: updateAccountSites soll NICHT reloadAllTimelines aufrufen**

    In `updateAccountSites` (Zeile 136-162), den Aufruf von `updateWidgetCredentials(for: updated)` (Zeile 155) ersetzen durch einen Aufruf der NUR die Daten schreibt, ohne `reloadAllTimelines`:

    Neue private Methode `syncWidgetData(for:)` einfuehren die nur schreibt:

    ```swift
    /// Schreibt Widget-Credentials und Account-Daten ohne Timeline-Reload
    private func syncWidgetData(for account: AnalyticsAccount) {
        let token: String
        switch account.providerType {
        case .umami:
            token = account.credentials.token ?? ""
        case .plausible:
            token = account.credentials.apiKey ?? ""
        }

        let sharedProviderType: SharedCredentials.ProviderType = account.providerType == .umami ? .umami : .plausible
        SharedCredentials.save(
            serverURL: account.serverURL,
            token: token,
            providerType: sharedProviderType,
            sites: account.sites
        )

        syncAccountsToWidget()
    }
    ```

    Die bestehende `updateWidgetCredentials(for:)` Methode BEHAELT `reloadAllTimelines()` — sie wird nur noch von `applyAccountCredentials` aufgerufen (nach allen async-Operationen).

    In `updateAccountSites` Zeile 155, aendern:
    VON: `updateWidgetCredentials(for: updated)`
    ZU: `syncWidgetData(for: updated)`

    **Fix 2: Sicherstellen dass in applyAccountCredentials der Widget-Reload NACH dem Notification-Post passiert**

    In `applyAccountCredentials` (Zeile 261-306) die Reihenfolge der letzten zwei Zeilen tauschen:

    VON (Zeile 302-305):
    ```swift
    updateWidgetCredentials(for: account)
    NotificationCenter.default.post(name: .accountDidChange, object: nil)
    ```

    ZU:
    ```swift
    // Erst Views benachrichtigen, dann Widget-Reload (alle Writes sind abgeschlossen)
    NotificationCenter.default.post(name: .accountDidChange, object: nil)
    updateWidgetCredentials(for: account)
    ```

    Das stellt sicher: alle async-Operationen (reconfigureFromKeychain, PlausibleSitesManager) sind fertig, Credentials sind in Keychain + SharedCredentials geschrieben, DANN erst reloadAllTimelines.

    **Problem:** `WebsiteDetailViewModel.loadData` verwendet `withTaskGroup` direkt — wenn loadData mehrfach aufgerufen wird (DateRange-Wechsel, Filter-Aenderung), laeuft die vorherige TaskGroup weiter. Beim Navigieren weg laufen die Tasks ebenfalls weiter (`.task{}` cancelt zwar, aber `onChange` und manuelle `Task{}` Aufrufe nicht).

    **Fix in WebsiteDetailViewModel.swift:**

    1. Neues Property `loadingTask` hinzufuegen nach `error` (Zeile 31):

    ```swift
    @Published var error: String?
    private var loadingTask: Task<Void, Never>?
    ```

    2. Die `loadData` Methode (Zeile 37-61) komplett ersetzen:

    ```swift
    func loadData(dateRange: DateRange) async {
        // Cancel vorherigen Load — verhindert Background-Battery-Drain (FIX-02)
        loadingTask?.cancel()
        let task = Task {
            isLoading = true
            defer {
                if !Task.isCancelled {
                    isLoading = false
                }
            }

            await withTaskGroup(of: Void.self) { group in
                group.addTask { await self.loadStats(dateRange: dateRange) }
                group.addTask { await self.loadActiveVisitors() }
                group.addTask { await self.loadPageviews(dateRange: dateRange) }
                group.addTask { await self.loadTopPages(dateRange: dateRange) }
                group.addTask { await self.loadPageTitles(dateRange: dateRange) }
                group.addTask { await self.loadReferrers(dateRange: dateRange) }
                group.addTask { await self.loadCountries(dateRange: dateRange) }
                group.addTask { await self.loadRegions(dateRange: dateRange) }
                group.addTask { await self.loadCities(dateRange: dateRange) }
                group.addTask { await self.loadDevices(dateRange: dateRange) }
                group.addTask { await self.loadBrowsers(dateRange: dateRange) }
                group.addTask { await self.loadOperatingSystems(dateRange: dateRange) }
                group.addTask { await self.loadLanguages(dateRange: dateRange) }
                group.addTask { await self.loadScreens(dateRange: dateRange) }
                group.addTask { await self.loadEvents(dateRange: dateRange) }
                group.addTask { await self.loadEntryPages(dateRange: dateRange) }
                group.addTask { await self.loadExitPages(dateRange: dateRange) }
                group.addTask { await self.loadGoals(dateRange: dateRange) }
            }
        }
        loadingTask = task
        await task.value
    }
    ```

    3. Neue `cancelLoading` Methode hinzufuegen (fuer explizites Cleanup):

    ```swift
    func cancelLoading() {
        loadingTask?.cancel()
        loadingTask = nil
    }
    ```

    **Fix in WebsiteDetailView.swift:**

    4. Den `.onChange(of: selectedDateRange)` Handler (Zeile 145-148) aendern — statt einen neuen unstrukturierten Task zu erzeugen, `.task(id:)` Modifier nutzen fuer automatische Cancellation:

    Zeile 142-148 ersetzen:
    VON:
    ```swift
    .task {
        await viewModel.loadData(dateRange: selectedDateRange)
    }
    .onChange(of: selectedDateRange) { _, newValue in
        Task {
            await viewModel.loadData(dateRange: newValue)
        }
    }
    ```

    ZU:
    ```swift
    .task(id: selectedDateRange) {
        await viewModel.loadData(dateRange: selectedDateRange)
    }
    ```

    Der `.task(id:)` Modifier cancelt automatisch den vorherigen Task wenn sich die ID aendert UND wenn die View verschwindet. Das ersetzt sowohl `.task{}` als auch den `.onChange` Handler.

    WICHTIG: Die anderen `Task { await viewModel.loadData(...) }` Aufrufe in den Filter-Handlern (Zeile 578, 730) koennen bestehen bleiben — das `loadingTask?.cancel()` in loadData() stellt sicher dass der vorherige Task gecancelt wird bevor der neue startet.
